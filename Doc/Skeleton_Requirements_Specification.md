## 1. Purpose / 目的

This document defines requirements for the NeoCEG Code Skeleton feature — a generator that outputs a virtual-language code skeleton from a `.nceg` file. The skeleton's logical structure mirrors the cause-effect graph, so that any implementation that follows the skeleton's structure achieves MC/DC coverage when tested with the decision table that NeoCEG generates.

本文書は NeoCEG コードスケルトン機能の要件を定義する。本機能は `.nceg` ファイルから仮想言語のコードスケルトンを生成するジェネレータであり、スケルトンの論理構造が原因結果グラフと同型となる。スケルトンの構造に従って実装される限り、NeoCEG が生成するデシジョンテーブルで MC/DC カバレッジが達成される。

---

## 2. Background and Rationale / 背景と根拠

### 2.1 The MC/DC Coverage Problem / MC/DC カバレッジの問題

The Cause-Effect Graphing technique generates test cases that cover each logical sub-expression of the graph independently. This is conceptually equivalent to MC/DC (Modified Condition/Decision Coverage) **on the graph**.

原因結果グラフ技法は、グラフの各論理部分式を独立にカバーするテストケースを生成する。これはグラフ上の MC/DC（Modified Condition/Decision Coverage）と概念的に等価である。

However, MC/DC is typically required of the **source code**, not of the specification. If the code's logical structure differs from the graph's logical structure (different short-circuit evaluation, different nesting of if-statements, factored-out helper functions, etc.), the test cases generated by NeoCEG do not necessarily achieve MC/DC on the code.

しかし、MC/DC は通常**ソースコード**に対して要求されるものであり、仕様に対するものではない。コードの論理構造がグラフの論理構造と異なる場合（短絡評価の違い、if 文の入れ子構造の違い、ヘルパー関数の切り出し等）、NeoCEG が生成するテストケースはコードに対する MC/DC を必ずしも達成しない。

### 2.2 Comparison with Code-Driven MC/DC Tools / コード駆動 MC/DC ツールとの比較

Code-driven MC/DC test generation tools (KLEE, CBMC, LDRA, VectorCAST, etc.) take the opposite approach.

コード駆動 MC/DC テスト生成ツール（KLEE、CBMC、LDRA、VectorCAST 等）は逆のアプローチを取る。

| Aspect / 観点 | Code-Driven MC/DC Tools / コード駆動ツール | NeoCEG (Specification-Driven) / NeoCEG（仕様駆動） |
|------|----------------|----------------|
| Input / 入力 | Source code / ソースコード | Specification (graph) / 仕様（グラフ） |
| Direction / 方向 | Code → Tests / コード → テスト | Specification → Tests → Code / 仕様 → テスト → コード |
| Analysis / 解析手法 | Symbolic execution, SMT solvers / シンボリック実行、SMT ソルバー | Logical expression enumeration / 論理式の組み合わせ列挙 |
| Test meaning / テストの意味 | Inputs that exercise the code / コードを通せる入力 | Inputs + expected outputs from spec / 仕様に基づく入力＋期待値 |
| Oracle / オラクル | Not provided / 持たない | Derived from specification / 仕様から導出 |

**The Oracle Problem / オラクル問題**:
Code-driven MC/DC tools have a fundamental limitation: they do not know the expected output. A test passes if the code executes without crashing. If the code contains a logic bug, the tool records the buggy behavior as "correct." Achieving 100% MC/DC coverage with such tools does not guarantee bug detection.

コード駆動 MC/DC ツールには根本的な制約がある — 期待される出力を知らない。コードがクラッシュせず実行できればテストは合格となる。コードに論理バグがあっても、ツールはバグを含んだ動作を「正しい」として記録してしまう。これらのツールで MC/DC 100% カバレッジを達成してもバグ検出は保証されない。

### 2.3 Why a Virtual-Language Skeleton / なぜ仮想言語スケルトンか

NeoCEG already provides specification-driven test design with proper expected values. The Code Skeleton feature aims to bridge the gap between specification and code structure, so that the MC/DC guarantee on the graph translates to MC/DC on the code.

NeoCEG は既に期待値を持つ仕様駆動テスト設計を提供している。コードスケルトン機能は、仕様とコード構造の間のギャップを埋め、グラフ上の MC/DC 保証をコード上の MC/DC へと変換することを目的とする。

**Why a virtual language rather than a concrete one / 具体的な言語ではなく仮想言語とする理由**:

1. **Avoid language-specific MC/DC quirks / 言語固有の MC/DC の罠を回避**
   Short-circuit evaluation (`&&`, `||`), implicit type conversion, null safety, exception propagation — each language has its own MC/DC interpretation. A virtual language with simple, total semantics avoids all of these.
   短絡評価、暗黙の型変換、null 安全、例外伝播 — 各言語ごとに MC/DC 解釈が異なる。単純で全域的な意味論を持つ仮想言語であれば、これらすべてを回避できる。

2. **One-to-one mapping with CEG semantics / CEG セマンティクスとの一対一対応**
   The virtual language's variables, AND/OR/NOT operators, and assignment correspond directly to CEG nodes, edges, and node definitions.
   仮想言語の変数、AND/OR/NOT 演算子、代入は CEG のノード、エッジ、ノード定義に直接対応する。

3. **Target-language independence / 対象言語からの独立性**
   The skeleton serves as a contract: "if you implement following this structure, MC/DC is guaranteed." Translation to C, Python, Java, etc. is the implementer's responsibility.
   スケルトンは契約として機能する — 「この構造に従って実装すれば MC/DC が保証される」。C、Python、Java 等への翻訳は実装者の責任となる。

4. **Documentation value / ドキュメントとしての価値**
   The skeleton can be embedded in design documents, code comments, or AI-assisted development prompts as a canonical representation of the logical structure.
   スケルトンは設計書、コードコメント、AI 支援開発のプロンプトに、論理構造の正規表現として埋め込める。

### 2.4 Constraint Strategy / 制約の扱い方の原則

A core design principle: **constraints are reflected in the input data structure, not in the function body.**

中核となる設計原則：**制約は入力データ構造に反映するものであり、関数本体に組み込まない。**

#### 2.4.1 Why constraints must not appear in the body / なぜ制約を本体に組み込まないか

If the body contains code that handles constraint violations (e.g., `if (A and B) error()` for an EXCL constraint), the test cases generated by NeoCEG — which only include constraint-satisfying inputs — will never exercise the violation branch. Such code paths cannot achieve MC/DC.

本体に制約違反を扱うコード（例：EXCL 制約のための `if (A and B) error()`）が含まれていると、制約を満たす入力のみを含む NeoCEG のテストケースでは違反分岐を通ることができない。そのコードパスは MC/DC を達成できない。

Therefore, constraints must be embedded in the input data structure such that violations cannot be expressed at all.

したがって制約は、違反そのものが表現できないような入力データ構造に埋め込む必要がある。

#### 2.4.2 Constraint-to-data-structure mapping / 制約とデータ構造の対応

| Constraint / 制約 | Data structure / データ構造 | Atomic condition form / 原子条件の形 |
|-------------------|---------------------------|----------------------------------|
| ONE(A, B, C) | Single variable / 単一変数 (one of A, B, C) | `var == "A"` |
| EXCL(A, B, C) | Single variable / 単一変数 (one of A, B, C, "none") | `var == "A"` |
| INCL(A, B, C) | Set / 集合 (non-empty subset of {A, B, C}) | `"A" in set` |
| REQ | Embedded in input structure / 入力構造に組込 | (no atomic form / 原子条件なし) |
| MASK | Comment only — naturally satisfied by expression structure / コメントのみ — 式構造により自然に成立 | (no atomic form / 原子条件なし) |

Rationale for INCL → set: when at least one of multiple options is true, the implementation typically processes each option independently. A set is the natural representation.

INCL → 集合の根拠：複数の選択肢のうち少なくとも1つが真である場合、実装ではそれぞれの選択肢を個別に処理することが多い。集合がその自然な表現となる。

#### 2.4.3 Optional constraint check function / 任意の制約チェック関数

When a redundant runtime verification of input constraints is desired (defense-in-depth, debugging, etc.), a separate check function can be generated as an option. The main body remains constraint-check-free to preserve MC/DC achievability.

入力制約の二重実行時検証が望ましい場合（多重防御、デバッグ等）、別関数として制約チェック関数をオプションで生成できる。MC/DC 達成可能性を維持するため、本体は引き続き制約チェックを含まない。

### 2.5 Limitations / 制約事項

This approach does not provide a fully-automated guarantee:

このアプローチは完全に自動化された保証を提供するものではない：

- The implementer must follow the skeleton's structure manually / 実装者がスケルトン構造に従うことは手動で担保される必要がある
- Refactoring the implementation to alter the logical structure breaks the MC/DC guarantee / 論理構造を変える実装リファクタリングは MC/DC 保証を破壊する
- Verifying structural conformance between skeleton and final implementation requires review or auxiliary tooling (out of scope for v1.0) / スケルトンと最終実装の構造的一致の検証はレビューまたは補助ツールを要する（v1.0 のスコープ外）

That said, **the alternative — independent specification and implementation — diverges far more**. The skeleton substantially reduces divergence by making the intended logical structure explicit and copyable.

ただし、**代替案である独立な仕様と実装は、はるかに大きく乖離する**。スケルトンは意図する論理構造を明示的かつコピー可能な形にすることで、乖離を実質的に大きく低減する。

## 4. System Requirements / システム要件

### 4.1 Virtual Language Definition / 仮想言語の定義

| ID | Requirement / 要件 | Parent / 親 |
|----|----------|--------|
| SKEL-SR-001 | Define a virtual language with one-to-one correspondence to NeoCEG DSL semantics / NeoCEG DSL セマンティクスと一対一対応する仮想言語を定義する | SKEL-UR-001 |
| SKEL-SR-002 | Publish the virtual language grammar as an EBNF specification / 仮想言語の文法を EBNF 仕様として公開する | SKEL-UR-003 |
| SKEL-SR-003 | Specify total (non-short-circuit) evaluation semantics for AND, OR, NOT / AND、OR、NOT に全域（非短絡）評価セマンティクスを規定する | SKEL-UR-002 |
| SKEL-SR-004 | Provide no side effects, exceptions, or implicit conversions / 副作用、例外、暗黙の変換を持たないものとする | SKEL-UR-002 |

#### Virtual Language Design / 仮想言語の設計

The virtual language uses **string values** for all data, with no type system beyond strings. This is the simplest representation that preserves logical structure without dragging in type-system complexity.

仮想言語はすべてのデータを**文字列値**として扱い、文字列以外の型システムを持たない。これは論理構造を保ちつつ型システムの複雑性を持ち込まない最も単純な表現である。

| Element / 要素 | Form / 形式 |
|---------------|-----------|
| Comment / コメント | `# ...` |
| Function declaration / 関数宣言 | `function NAME(params) -> (returns) { body }` |
| Parameter / 引数 | `name` (with optional `# allowed values` comment) |
| Assignment / 代入 | `name := expression` |
| Logical operators / 論理演算子 | `AND`, `OR`, `NOT` |
| Equality / 等価比較 | `==` |
| Set membership / 集合包含 | `in` |
| Grouping / グルーピング | `( ... )` |

If string typing proves insufficient, the design will be reconsidered.

文字列型で不都合が出てきた場合は設計を見直す。

## 5. Non-Functional Requirements / 非機能要件

### 5.1 Compatibility / 互換性

| ID | Requirement / 要件 |
|----|----------|
| SKEL-NF-001 | Reuse the existing DSL parser and logical model / 既存の DSL パーサーと論理モデルを再利用する |
| SKEL-NF-002 | Produce identical skeletons for identical inputs (deterministic) / 同一入力に対し同一スケルトンを生成する（決定的） |

### 5.2 Design Constraints / 設計制約

| ID | Constraint / 制約 |
|----|----------|
| SKEL-NF-010 | The virtual language must remain executable in concept (a reasonable translation to C/Python/Java should be straightforward) / 仮想言語は概念的に実行可能であること（C/Python/Java への合理的な翻訳が容易であること） |
| SKEL-NF-011 | The skeleton must be human-readable; it is meant to be read by developers and AI assistants / スケルトンは人間可読であること（開発者および AI アシスタントによる読み取りを想定） |
| SKEL-NF-012 | The observable flag is not currently reflected in the skeleton (the flag is unused by the algorithm and may be deprecated) / observable フラグは現状スケルトンに反映しない（アルゴリズムで未使用であり、将来撤廃の可能性がある） |

## 6. Example / 例

The following example illustrates the skeleton output for an admission fee graph with three ONE constraints.

以下は3つの ONE 制約を持つ入場料金グラフに対するスケルトン出力例：

```
# Generated from: Admissionfee_jp.nceg
# Cause-Effect Graph: 入場料金計算

function admission_fee(
  category,       # individual | group         (from ONE constraint)
  age_group,      # 65plus | adult | elementary | under6
  residence       # resident | non_resident
) -> (free, fee_1200, fee_1000, fee_600, fee_500) {

  # Intermediate nodes
  resident_elementary := (age_group == "elementary") AND (residence == "resident")
  non_resident_elementary := (age_group == "elementary") AND (residence == "non_resident")

  # Effects
  free := ((category == "individual") AND (age_group == "65plus") AND (residence == "resident"))
       OR ((category == "group") AND (age_group == "under6"))
       OR resident_elementary

  fee_1200 := (category == "individual") AND (age_group == "adult") AND (residence == "resident")
  fee_1000 := (category == "group") AND (age_group == "adult") AND (residence == "non_resident")
  fee_600  := non_resident_elementary
  fee_500  := (category == "group") AND (age_group == "elementary") AND (residence == "non_resident")
}
```
